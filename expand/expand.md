## Part 3 Answers

1. Honestly, I think the asynchronous nature of JavaScript is something most developers struggle with at some point, because your code doesn't execute in the order you write it, and until you really understand the event loop, you'll find yourself confused about why certain values aren't available when you expect them to be. Loose typing is another pain point I feel like gets underestimated, because JavaScript is so permissive that it will silently coerce types without warning you, meaning bugs can hide in your codebase for a long time before you even realize something went wrong since the language never stops you and says "hey, this doesn't make sense." And then the web platform itself, in my opinion, adds a unique kind of frustration because JavaScript evolved alongside the web rather than being designed cleanly from the ground up, so developers are still dealing with browser inconsistencies, legacy APIs, and constraints that were baked in decades ago and can't really be removed without breaking half the internet.

2. I think JavaScript was made loosely typed because it was never originally intended for large scale applications, it was meant to be a simple scripting language that designers and hobbyists could pick up quickly without needing a computer science background, so strict typing would have just gotten in the way of that. As for asynchronous features, I feel like that was more of a practical necessity than a conscious design choice, because JavaScript runs in the browser and the browser is constantly waiting on things like network requests and user input, so if everything ran synchronously the page would freeze every time it had to fetch data, which would have made for a pretty terrible user experience.

3. A compiled language gets translated into machine code all at once before it ever runs, while an interpreted language gets read and executed line by line at runtime, and I think the key difference you feel as a developer is that compiled languages tend to catch errors early and run faster, while interpreted languages are more flexible and easier to get up and running quickly. JavaScript is interpreted, and in my opinion that's a big part of why it took off the way it did, because you don't need a build step, you just write code and the browser runs it instantly. The drawback though is that errors only show up at runtime rather than being caught ahead of time, and interpreted languages are generally slower than compiled ones, which for most web use cases is fine but can become a real limitation if you're doing something computationally heavy.

4. I think the professor is focusing on vanilla JavaScript first because frameworks are really just abstractions built on top of the language itself, so if you don't understand what's happening underneath, you're essentially just memorizing syntax without really knowing why things work the way they do. In my opinion, mastering vanilla JS gives you a much stronger foundation because you actually understand the core concepts like DOM manipulation, event handling, and asynchronous behavior rather than relying on a framework to handle it for you invisibly. The drawback of skipping frameworks though is that in the real world most teams are using React or Vue or Angular, so there is a bit of a gap between what you learn in a vanilla JS course and what you'll encounter on the job, but I feel like that gap is a lot easier to close once you actually understand the language deeply rather than the other way around.
